# M5 目录 Meta、隐藏规则与下载签名设计

> 状态：待批准。批准后再进入编码。
> 分支：`feat(meta)/m5-meta-sign`（实现阶段从 M4 基线切出）。

## 1. 解决的问题

M4 完成后已是可用的最小网盘，但缺少 AList 的目录治理能力：目录密码、隐藏规则、README/Header 展示，以及下载链接签名。M5 补齐这四项，使目录可被精细化控制，并为 M7 公开分享的免登录下载预埋签名机制。

不在本阶段范围：分享（M7）、任务/索引（M6）、AI（M9）。M5 的 `/d/**` 仍是**登录态**接口，签名只作为附加校验；真正的免登录公开下载在 M7。

## 2. 为什么这样选

- **复用既有表与字段**：`meta_rules` 表（M1 已建）、`FsListResponse.readme/header/write`、`FileObjectResponse.sign`、`FsListRequest.password` 均已存在，本阶段只需填充，不改 DB schema、不加迁移脚本。
- **复用既有权限位**：`PermissionBits.VIEW_HIDDEN`（看隐藏项）、`BYPASS_PASSWORD`（免目录密码）已定义，直接作为隐藏过滤与密码豁免的判定，无需新增权限模型。
- **Admin 接口自动受保护**：`AuthenticationInterceptor` 对 `/api/admin/**` 强制 admin，`/api/admin/meta/**` 天然纳入，无需额外守卫。
- **签名不引依赖**：HMAC-SHA256 用 JDK `javax.crypto.Mac` 实现，密钥复用 `asuka.jwt.secret` 或新增独立配置，零新依赖。
- **沿用分层**：Meta CRUD 走 `AdminMetaController → MetaApplicationService → MetaRuleMapper`，与 Storage 模块同构。

## 3. 总体架构

```
Meta 管理:  AdminMetaController(/api/admin/meta/**) ─► MetaApplicationService ─► MetaRuleMapper

list 读路径治理:
  FsApplicationService.list
     ├─ MetaResolveService.resolve(internalPath) ─► ResolvedMeta(password/hide/readme/header/writeEnabled)
     ├─ 密码校验（无 BYPASS_PASSWORD 时必须匹配，否则 PASSWORD_REQUIRED/INCORRECT）
     ├─ 隐藏过滤（无 VIEW_HIDDEN 时按 hide 正则剔除条目）
     ├─ 填充 readme/header/write 到 FsListResponse
     └─ 为文件条目填充 DownloadSignService.sign(visiblePath)

下载:  DownloadController(/d/**)
     ├─ needsSign(internalPath) = signAll || 该路径命中带密码的 meta
     └─ 需要签名且非 admin ⇒ 校验 ?sign= → 通过后走 FsApplicationService.link（仍校验读权限）
```

## 4. 实现方式（分提交批次）

> 每批一个或多个 commit，逐步可编译可测。

**批次 1 `feat(meta)`：Meta 领域 + CRUD**
- `MetaRule` / `ResolvedMeta` 领域对象；`MetaModelMapper`。
- `MetaApplicationService`：`list/get/create/update/delete`，`create/update` 校验 path 规范化与唯一、hide 正则可编译。
- `AdminMetaController` + Request/Response DTO。

**批次 2 `feat(meta)`：路径就近解析**
- `MetaApplicationService.resolve(internalPath)`：加载该路径的祖先-或-自身 meta，按属性就近生效（自身命中全部生效；祖先命中仅当对应 `*_sub=true`）。返回 `ResolvedMeta`。
- 小表策略：`selectList` 全量后在内存按前缀匹配（数据量小，AList 同样全量缓存）；后续可加缓存。

**批次 3 `feat(fs)`：list/get 接入治理**
- 密码：无 `BYPASS_PASSWORD` 且 meta 有密码时，`request.password` 缺失 → `PASSWORD_REQUIRED`，不匹配 → `PASSWORD_INCORRECT`。
- 隐藏：无 `VIEW_HIDDEN` 时，按 hide 正则逐行过滤条目（仅过滤展示，不影响直接 get/下载——与 AList 一致，隐藏≠鉴权）。
- README/Header/write：填充 `FsListResponse`；`write` = 解析权限含写位 **或** meta `writeEnabled` 生效。

**批次 4 `feat(fs)`：下载签名**
- `DownloadSignService.sign(path)` → `base64url(hmac(path:exp)) + ":" + exp`；`verify(path, sign)` 校验 hmac 与过期。
- `list/get` 为文件条目填充 `sign`（目录不签名）。
- `DownloadController`：`needsSign` 为真且非 admin 时校验 `?sign=`，失败 `PERMISSION_DENIED`；其余维持 M4 登录态读权限行为（向后兼容，默认 `signAll=false`）。

**批次 5 `test`/`feat(web)`/`docs`：测试 + 前端 + 文档**
- 单测见 §7；前端目录密码弹窗 + README/Header 渲染 + 隐藏项随后端生效；更新 README API 表与本文 §7。

## 5. 关键接口与数据流

### 接口签名（HTTP）

| 方法 | 路径 | 权限 | 说明 |
| --- | --- | --- | --- |
| `GET` | `/api/admin/meta/list` | admin | meta 列表 |
| `GET` | `/api/admin/meta/get?id=` | admin | meta 详情 |
| `POST` | `/api/admin/meta/create` | admin | 创建 meta |
| `POST` | `/api/admin/meta/update` | admin | 更新 meta |
| `POST` | `/api/admin/meta/delete` | admin | 删除 meta |
| `POST` | `/api/fs/list` | 读（+目录密码）| 列表，接入隐藏/密码/readme/header/sign |
| `GET` | `/d/**?sign=` | 读（需签名时校验）| 下载 |

### 就近 Meta 解析（resolve）

```
输入: internalPath（已拼 basePath 的内部路径）
1. 取所有 path 为 internalPath 的祖先-或-自身的 meta
2. 对每个属性（password/hide/readme/header/write）：
   - 自身精确命中：直接生效
   - 祖先命中：仅当该属性的 *_sub = true 才生效
   - 多个候选取“最近”（路径最长）的那个
3. 返回 ResolvedMeta（任一属性可能为空）
```

### list 密码 + 隐藏数据流

```
1. 解析 internalPath，读权限校验（沿用 M4）
2. meta = resolve(internalPath)
3. 若 meta.password 非空且用户无 BYPASS_PASSWORD:
     password 缺失 → PASSWORD_REQUIRED
     password 错误 → PASSWORD_INCORRECT
4. 驱动列目录
5. 若用户无 VIEW_HIDDEN: 按 meta.hide 正则剔除匹配条目
6. 文件条目填充 sign；响应填充 readme/header/write
```

### 下载签名数据流（/d/**）

```
1. extractPath → visiblePath；解析 ?sign=
2. needsSign = signAll || resolve(internalPath).password 非空
3. needsSign 且 !admin:
     verify(visiblePath, sign) 失败 → PERMISSION_DENIED
4. fsApplicationService.link（内部仍按登录用户校验读权限）→ 回写文件流
```

> 签名绑定**用户可见路径**（与前端 `/d/<path>?sign=` 一致）。M5 下载仍登录态，基于登录用户 basePath 做权限校验，故签名只需绑定可见路径与过期时间；M7 分享场景再扩展为绑定分享上下文。

## 6. 异常处理

| 场景 | 错误码 | HTTP |
| --- | --- | --- |
| 目录需密码但未提供 | `PASSWORD_REQUIRED`（新） | 403 |
| 目录密码错误 | `PASSWORD_INCORRECT`（新） | 403 |
| meta path 非法 / hide 正则不可编译 | `BAD_REQUEST` | 400 |
| meta path 已存在 | `BAD_REQUEST` | 400 |
| meta 不存在（get/update/delete）| `OBJECT_NOT_FOUND` | 404 |
| 需签名但 sign 缺失/无效/过期 | `PERMISSION_DENIED` | 403 |
| 非 admin 访问 `/api/admin/meta/**` | `PERMISSION_DENIED` | 403 |

> 前端：密码类错误用 HTTP 403 + 独立 `code`（不用 401，避免被 axios 拦截器误判为登出）；前端据 `code` 弹出密码框。

## 7. 风险、限制、后续（测试待实现后回填）

- **限制**：meta 密码为明文存储（沿用 schema 与 AList 行为），适用于低敏目录口令；哈希化留作后续。
- **限制**：隐藏规则仅过滤列表展示，不阻止已知路径的直接 `get`/下载（与 AList 一致，隐藏≠鉴权）。
- **限制**：M5 `/d` 仍登录态；签名绑定可见路径，跨用户同名可见路径不互通靠登录态权限兜底，公开分享在 M7 完善。
- **风险**：meta 全量内存匹配，超大 meta 量下需加缓存；当前数据量小可接受。
- **后续**：meta 变更后的缓存失效、签名绑定分享上下文（M7）、目录密码哈希化。
